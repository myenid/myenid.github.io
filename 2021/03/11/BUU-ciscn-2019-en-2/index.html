主流程分析

libc泄露，过程不分析了，贴源码上来。

int encrypt()
{
  size_t v0; // rbx
  char s[48]; // [rsp+0h] [rbp-50h]
  __int16 v3; // [rsp+30h] [rbp-20h]

  memset(s, 0, sizeof(s));
  v3 = 0;
  puts("Input your Plaintext to be encrypted");
  gets(s);
  while ( 1 )
  {
    v0 = (unsigned int)x;
    if ( v0 >= strlen(s) )
      break;
    if ( s[x] <= '`' || s[x] > 'z' )
    {
      if ( s[x] <= '@' || s[x] > 'Z' )
      {
        if ( s[x] > '/' && s[x] <= '9' )
          s[x] ^= 0xCu;
      }
      else
      {
        s[x] ^= 0xDu;
      }
    }
    else
    {
      s[x] ^= 0xEu;
    }
    ++x;
  }
  puts("Ciphertext");
  return puts(s);
}

这是漏洞函数，看起来是要加密，实际上利用strlen的漏洞可以绕过加密。

要点

recv

我每次觉得泄露libc都是玄学

这次还是一样的，在这里对libc泄露字节的接受做一个总结吧

  • 首先，函数返回之前不会puts(x)，这样最简单，直接先recvuntil(xxx)，然后直接recvuntil(‘\n’)[-1:].ljust(8,’\x00’)
  • 然后如果有pust这就比较复杂了，目前我最喜欢的是，先计算字节，然后recvuntil(‘\x7f’)[-6:].ljust(8,’\x00’)ps:这是64位的
  • 32位就recvuntil(‘\xf7’)就行
  • 这样不行的话，现在又会了一个p.recvline()这个函数，一般recv()字节不行的话，0xa这个换行符很讨厌，建议debug然后过不了的话就尝试下这个吧

strlen

然后就是万恶的strlen()函数

怎么截断都懂，但是用不起来。。。

自己本地复制了漏洞代码，结果发现编译器太高级，自动把\转义。。。，导致写payload不敢下手。这里也当学会了strlen吧。

栈平衡

写这个payload发现，高级的libc二次利用，或者利用代码的时候，加上了一个ret，因为，高版本libc在运行的时候，会验证栈的位置。如果不是0x10的倍数就会拒绝执行，这里的话可能刚好就不是，导致了总是运行超时，拿不到shell, 0x4197507这是pop_rdi的地址，果然需要平衡。
